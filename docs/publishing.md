# Publishing

This repo is primarily distributed via **Git clone or `cp -r`**, not npm. That said, we want every skill to be ready for registry publication the day we choose to.

## Distribution today

A user installs a skill from this repo by one of:

```bash
# Whole repo
git clone https://github.com/<owner>/claude-skills

# Single skill (sparse checkout)
git clone --filter=blob:none --no-checkout https://github.com/<owner>/claude-skills
cd claude-skills
git sparse-checkout init --cone
git sparse-checkout set skills/<skill-name>
git checkout

# Or just download the folder
curl -L https://github.com/<owner>/claude-skills/archive/main.tar.gz | tar xz
cp -r claude-skills-main/skills/<skill-name> ~/.claude/skills/
```

The skill is then dropped into the user's agent's skills directory (e.g., `~/.claude/skills/` for Claude Code) and is immediately available.

## What makes a skill "publish-ready"

Even though we don't publish today, every skill should pass this checklist:

- [ ] `SKILL.md` frontmatter valid (`pnpm lint` passes for this skill)
- [ ] `README.md` describes purpose, usage, and limitations
- [ ] `package.json` has a `description`, `license`, and `version`
- [ ] No hardcoded paths to maintainer's machine
- [ ] No leaked credentials or API keys
- [ ] No references to other skills in this monorepo (unless declared in `metadata.linked-skills`)
- [ ] `examples/` contains at least one minimal usage example
- [ ] License is declared (defaults to repo MIT unless skill overrides)

`pnpm health` runs these checks across all skills.

## When publishing happens

We will add registry publication when **at least three** of these are true:

- A skill has external users who request pinned versions
- Manual installation friction is a real complaint
- We can commit to triaging security advisories on a public registry
- A consumer registry (skills.sh, agentskills.io marketplace) becomes the dominant install path
- We want SLSA / sigstore provenance for trust

Until then, "clone the repo" is fine.

## When that happens, the order of operations is

1. Add Changesets (see `docs/tooling.md`)
2. Decide on package scope (`@<scope>/<skill-name>`)
3. Decide registry (npm, skills.sh, both)
4. Set up `prepublishOnly` validation in each skill's `package.json`
5. Configure CI to publish on changeset merges
6. Add `--provenance` flag if using npm
7. Update `docs/publishing.md` (this file) with the live process

## Versioning policy (when we publish)

- Each skill has its own SemVer line.
- Breaking change to instructions = major bump (because consumers' agents may behave differently)
- New optional capability = minor bump
- Wording fix, typo, internal restructure that doesn't change agent behaviour = patch bump
- Removing a field from `forbidden-behaviours` is a major bump (it loosens the contract)

## Provenance & trust

When we publish, we will:

- Publish with npm `--provenance` (sigstore-backed attestation)
- Sign git tags
- Maintain a `.well-known/skills.json` at the repo root with hashes of each shipped skill version

## Skill discoverability

Independent of npm publishing, every skill should be discoverable via:

- The root `SKILLS.md` index (regenerated by `pnpm index`)
- The GitHub topic `claude-skill`
- Submission to https://agentskills.io once we have a stable URL per skill
